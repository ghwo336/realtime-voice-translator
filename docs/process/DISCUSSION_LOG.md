## 2026-05-26 (Day 3, 화) — 요구 모델링 (Phase 2 종료) ✅ 완료

### 토의 주제 1. 다이어그램 표현 방식
- **검토**: PNG 이미지(draw.io/PlantUML 렌더) vs **Mermaid 텍스트 다이어그램**
- **결정**: **Mermaid 채택**
- **근거**: (a) GitHub가 .md 안의 Mermaid를 자동 렌더 → 채점자가 도구 없이 즉시 시각화 확인 가능, (b) git diff로 변경점이 보임 (이미지는 안 보임), (c) 텍스트라 형상관리 친화적.

### 토의 주제 2. 유스케이스 도출
- **방법**: 강의록 04 §4.4 슬라이드 25의 5개 질문에 답하는 방식으로 UC 후보 추출
- **결과**: UC-01 ~ UC-07 (7건). 액터 5종 (User + 외부 시스템 4종).
- **관계**: include 2건(UC-01→UC-04, UC-04→UC-05), extend 1건(UC-02→UC-07).

### 토의 주제 3. 클래스 후보 식별 방법
- **검토**: 명사 추출법 vs CRC 카드 vs 도메인 사전
- **결정**: **요구사항 명세서·시나리오의 명사**에서 클래스 후보 추출 (강의록 05 §5.2 모델링 과정 ②)
- **결과**: 11개 클래스 후보 + 4개 후보 탈락(중복/액터/세부 구현)

### 토의 주제 4. 외부 시스템을 `<<interface>>`로 모델링한 이유
- **결정**: STT/Translator/AudioCapture/Diarizer/LogStorage를 모두 인터페이스로
- **근거**: Open Issues O-01 ~ O-04 (기술 스택)가 미정. **인터페이스로 캡슐화**해두면 Day 4의 기술 선택이 클래스 다이어그램을 바꾸지 않음. 강의록 05 §5.3 캡슐화 원칙 적용.
- **교훈 (lessons learned 후보 #2)**: **"바이브코딩에서는 외부 컴포넌트가 자주 교체된다 (AI가 제안하는 라이브러리가 바뀜). 따라서 도메인 모델은 처음부터 인터페이스로 캡슐화하는 것이 유지보수 비용을 크게 낮춘다."**
