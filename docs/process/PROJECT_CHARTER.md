## 2. 동기 및 가치 (Why this project)

### 2.1 문제 인식
영어 회의·강의·인터뷰를 들을 때 다음 두 가지 어려움이 동시에 발생한다.
1. **언어 장벽** — 실시간으로 의미를 따라가기 어려움
2. **화자 식별** — 여러 사람이 번갈아 말할 때 "누가 무엇을 말했는지" 추적이 어려움

기존 도구는 (a) 단순 STT (Otter 등) 또는 (b) 단순 번역(Google Translate 마이크 입력)에 머무르며, 두 기능을 결합해 **"화자 A: ... / 화자 B: ..."** 형태로 한국어 자막을 제공하는 도구는 일반 사용자가 쉽게 쓸 수 있는 무료 데스크탑 형태로 존재하지 않는다.

### 2.2 기대 가치
- **개인 가치**: 본인이 실제 영어 콘텐츠 학습·회의 참여 시 사용 가능
- **학습 가치**: STT/화자분리/번역/실시간 UI 동기화 등 다양한 기술적 도전이 결합돼 있어 **요구사항 분석 ~ 품질관리까지 프로세스 전 단계의 의미있는 적용 사례**가 됨

## 3. 범위 (Scope)

### 3.1 In Scope (이번 프로젝트에서 다룸)
- 마이크 입력 또는 시스템 오디오 캡처 → 영어 음성 인식
- 2~4명 화자의 자동 분리 (간이 diarization)
- 영어 → 한국어 번역
- 화자별로 색상/라벨이 구분된 자막 UI
- 세션 종료 후 대화 로그 텍스트 파일로 저장
- 윈도우/맥OS 데스크탑 환경

### 3.2 Out of Scope (이번 프로젝트에서 제외)
- 모바일 앱 (iOS/Android)
- 한→영 번역, 기타 언어 쌍
- 음성 합성(TTS)으로 한국어 발화 재생
- 화자 등록(누가 A인지 사전 학습)
- 클라우드 동기화, 사용자 계정

## 5. 프로세스 모델 선정

### 5.1 후보 비교 (강의록 02 기반)

| 모델 | 적합도 | 평가 |
|---|---|---|
| 폭포수 | 보통 | 단계가 명확하나 "테스트가 후반에 시작", "변경 수용 불가" 단점이 바이브코딩과 충돌 |
| V 모델 | 보통 | 검증 강화 관점은 좋으나 단일 사이클이라 위험관리 약함 |
| 프로토타이핑 | 좋음 | 요구 불투명·기술 실현 가능성 타진에 적합 — RVT의 STT/diarization이 여기 해당 |
| **나선형** | **매우 좋음** | "재정적 또는 기술적 위험 부담이 큰 경우" 적합. 점증적 릴리스. |
| Unified Process | 좋음 | 잘 문서화되나 1인 프로젝트엔 복잡 |
| 애자일 (Scrum/XP) | 좋음 | 짧은 주기 반복은 매력적이나 "팀 협업" 관점이 1인에는 약함 |

### 5.2 선정: **하이브리드 (Waterfall 단계 골격 + 나선형 반복 구현)**

**근거**:
1. **교과서 표준 단계 명세성** 확보 — 요구→설계→구현→테스트의 단계별 산출물을 분명히 남길 수 있다 (폭포수의 장점).
2. **기술적 위험 통제** — STT 정확도, 화자분리 신뢰도, 실시간성 등은 사전에 예측 불가. 구현 단계를 **2회 나선형 사이클(Sprint 1, 2)**로 나눠 각 사이클 시작에 위험 분석을 수행.
3. **변경 수용** — 1차 사이클 결과를 기반으로 2차에서 요구·설계를 재조정.

### 5.3 사이클 구성

```
[Phase 1: Inception]      Day 1       프로젝트 헌장 (이 문서)
[Phase 2: Requirements]   Day 2-3     SRS + 유스케이스
[Phase 3: Design]         Day 4-5     아키텍처 + UML + UI 와이어프레임
[Phase 4: Construction]   Day 6-10    Sprint 1 (Day 6-7): STT + 번역 단방향 PoC
                                      Sprint 2 (Day 8-10): Diarization + UI 통합
[Phase 5: Testing & QA]   Day 11-12   단위/통합 테스트, 결함 로그
[Phase 6: Closure]        Day 13-14   보고서, PDF 변환, 최종 제출
```
