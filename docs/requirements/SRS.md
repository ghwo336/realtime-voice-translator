## 1. 서론 (Introduction)

### 1.1 목적
본 문서는 Real-time Voice Translator의 기능 요구사항(FR), 비기능 요구사항(NFR), 제약사항을 명세하여 후속 설계·구현·테스트의 단일 진실 원천(single source of truth)이 되는 것을 목적으로 한다.

### 1.2 범위 (Scope)
Charter §3 (In/Out Scope)와 동기화된다.
- **포함**: 영→한 다중 화자 실시간 음성 번역, Mac 데스크탑, 로컬 텍스트 로그 저장
- **제외**: 한→영 / 모바일 / TTS / 사용자 계정 / 클라우드 동기화

### 1.3 정의
| 용어 | 정의 |
|---|---|
| 발화(utterance) | 한 화자가 무음으로 끊어지지 않고 이어서 말하는 한 단위의 음성 |
| 세션(session) | "시작" 버튼부터 "종료" 버튼까지의 단일 사용 단위 |
| 청크(chunk) | STT 처리를 위한 오디오의 시간 분할 단위 (예: 1초) |
| 화자 라벨 | 화자 분리 결과로 자동 부여되는 식별자 (Speaker-A, Speaker-B …) |

### 1.4 참조 문서
- 강의록 04_요구.ppt, 05_요구 모델링.ppt
- 본 repo: PROJECT_CHARTER.md, USER_SCENARIOS.md
- IEEE 830-1998 표준 (요구 명세 양식)

## 2. 전체 설명 (Overall Description)

### 2.1 제품 관점
RVT는 독립 실행형 데스크탑 애플리케이션이다. 외부 의존성은 (a) 운영체제 오디오 API, (b) STT 엔진(로컬 또는 클라우드), (c) 번역 엔진(로컬 또는 클라우드)이며, 각 의존성은 설계 단계(Day 4-5)에서 선택된다.

### 2.2 사용자 클래스
| 클래스 | 특성 | 빈도 |
|---|---|---|
| 주 사용자 | 영어 회의 참여자, 한국어 모국어, 기술적 친숙도 중상 | 주 3~5회 |
| (Out of scope) 화자 등록자 | 본 버전에서는 없음 | - |

### 2.3 운영 환경
- macOS 12 이상 (1순위)
- 마이크 또는 가상 오디오 입력(BlackHole, Loopback 등) 접근 가능
- 인터넷 연결 가능 (클라우드 STT/번역 채택 시)

### 2.4 가정 및 의존성
- 사용자는 회의 시작 전 입력 디바이스를 인지하고 선택할 능력이 있다.
- 회의 화자 수는 2~4명 수준 (다섯 명 이상은 화자 분리 정확도가 급락할 수 있음 — Risk R2 참조)

## 3. 기능 요구사항 (Functional Requirements, FR)

> **표기 규칙**: 강의록 04 §4.1 "‘시스템은 ~을 해야 한다’" 양식 준수.
> 각 FR은 [원자성/테스트 가능성] 원칙에 따라 단일 검증 가능한 행위로 기술한다.

### FR-001 오디오 입력 캡처
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 사용자가 선택한 오디오 입력 디바이스에서 PCM 오디오 스트림을 캡처해야 한다. |
| 출처 | S-01, S-03 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 단위 테스트: mock 디바이스로 1초 캡처 후 비-zero 데이터 확인 |

### FR-002 영어 음성 인식 (STT)
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 캡처된 영어 오디오 청크를 텍스트로 변환하여 영어 원문을 산출해야 한다. |
| 출처 | S-01 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 통합 테스트: 표준 영어 샘플 오디오 → 단어 오류율(WER) 측정 |

### FR-003 영→한 번역
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 STT 결과 영어 텍스트를 한국어로 번역하여 자막에 표시해야 한다. |
| 출처 | S-01 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 통합 테스트: 사전 정의된 영어 문장 10개 → 번역 결과의 의미 보존 여부 수동 평가 |

### FR-004 화자 분리 (Diarization)
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 2~4명 화자의 발화를 자동으로 분리하여 각 발화에 화자 라벨(Speaker-A, B …)을 부여해야 한다. |
| 출처 | S-01 |
| 우선순위 | 필수 (MVP). Risk R2에 따라 Sprint 1 실패 시 "에너지 기반 단순 분리"로 fallback |
| 검증 방법 | 통합 테스트: 2명 화자 샘플로 화자 분리 정확도(DER) 측정 |

### FR-005 화자 라벨 일관성
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 한 세션 내에서 동일 화자에게 동일 라벨을 일관되게 유지해야 한다. |
| 출처 | S-01 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 통합 테스트: 5분 샘플에서 화자 전환 직후 라벨 변경 빈도가 5% 이하 |

### FR-006 세션 로그 자동 저장
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 세션 종료 시 화자별 대화 로그를 사용자 지정 또는 기본 폴더(`~/Documents/RVT/`)에 자동 저장해야 한다. |
| 출처 | S-02 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 통합 테스트: 세션 종료 후 파일 존재 + 내용 비-empty 확인 |

### FR-007 로그 포맷
| 항목 | 내용 |
|---|---|
| 설명 | 저장된 로그는 각 발화에 대해 (a) 타임스탬프, (b) 화자 라벨, (c) 영어 원문, (d) 한국어 번역을 줄 단위로 포함해야 한다. |
| 출처 | S-02 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 단위 테스트: 합성 발화 3건 입력 → 출력 파일 정규식 매칭 |

### FR-008 입력 디바이스 선택 및 검증
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 사용 가능한 오디오 입력 디바이스 목록을 표시하고, 선택된 입력에서 신호가 감지되어야만 "시작" 버튼이 활성화되도록 해야 한다. |
| 출처 | S-03 |
| 우선순위 | 중요 |
| 검증 방법 | UI 테스트: 무음 입력 → 버튼 비활성, 신호 입력 → 활성 |

### FR-009 오류 통지
| 항목 | 내용 |
|---|---|
| 설명 | STT 또는 번역 호출 실패 시 시스템은 자막 영역에 가시적 오류 표시("⚠ 번역 일시 중단")를 하고, 5초 간격으로 자동 재시도해야 한다. |
| 출처 | S-04 |
| 우선순위 | 권장 |
| 검증 방법 | 통합 테스트: 번역 API mock을 강제 실패 → UI에 오류 표시, 재시도 호출 검증 |

### FR-010 세션 제어 (시작/일시정지/종료)
| 항목 | 내용 |
|---|---|
| 설명 | 시스템은 시작·일시정지·종료의 세 가지 상태 전환 버튼을 제공해야 한다. |
| 출처 | S-01, S-02 |
| 우선순위 | 필수 (MVP) |
| 검증 방법 | 상태 전환 매트릭스 테스트 |

## 4. 비기능 요구사항 (Non-Functional Requirements, NFR)

> **표기 규칙**: 강의록 04 §4.3 "테스트 가능성" 원칙에 따라 가능한 한 정량적 기준을 명시.

### NFR-001 지연 (Latency)
| 항목 | 내용 |
|---|---|
| 설명 | 발화 종료 시점부터 한국어 자막 표시까지의 평균 지연은 3초 이하여야 하며, 95퍼센타일에서 5초를 초과하지 않아야 한다. |
| 출처 | S-01 |
| 우선순위 | 필수 |
| 검증 방법 | 성능 테스트: 표준 샘플 30개 발화에 대한 지연 측정, 통계 산출 |

### NFR-002 번역 정확도
| 항목 | 내용 |
|---|---|
| 설명 | 표준 비즈니스 회화 샘플 50문장에 대해 번역 결과의 "의미 보존 평가"(주관 평가, 5점 척도)에서 평균 4.0 이상이어야 한다. |
| 출처 | S-01 |
| 우선순위 | 필수 |
| 검증 방법 | 수동 평가 + 사후 BLEU/COMET 보조 측정 |

### NFR-003 STT 정확도
| 항목 | 내용 |
|---|---|
| 설명 | 깨끗한 영어 회의 샘플(LibriSpeech test-clean 등)에서 WER ≤ 10%여야 한다. |
| 출처 | S-01 (암묵) |
| 우선순위 | 중요 |
| 검증 방법 | LibriSpeech 일부 샘플 평가 |

### NFR-004 사용성 (Usability)
| 항목 | 내용 |
|---|---|
| 설명 | 처음 사용자가 매뉴얼 없이 앱 실행 후 60초 이내에 "시작" 버튼까지 도달할 수 있어야 한다. |
| 출처 | S-03 |
| 우선순위 | 중요 |
| 검증 방법 | 사용성 테스트 (자기 검증 + 외부 1인 검증 권장) |

### NFR-005 신뢰성 (Reliability)
| 항목 | 내용 |
|---|---|
| 설명 | 단일 컴포넌트(STT 또는 번역)의 일시적 실패가 세션 전체 종료로 이어지지 않아야 한다. 부분 실패 시 다른 컴포넌트는 계속 동작해야 한다. |
| 출처 | S-04 |
| 우선순위 | 중요 |
| 검증 방법 | 통합 테스트: 컴포넌트별 fault injection |

### NFR-006 플랫폼 통합
| 항목 | 내용 |
|---|---|
| 설명 | macOS 12+ 에서 마이크 권한 요청이 표준 OS 다이얼로그로 자연스럽게 처리되어야 한다. |
| 출처 | S-03 |
| 우선순위 | 필수 |
| 검증 방법 | 클린 환경에서 첫 실행 시 권한 다이얼로그 노출 확인 |

### NFR-007 자원 사용
| 항목 | 내용 |
|---|---|
| 설명 | 일반 회의 사용 중(2~4명 화자, 1시간) CPU 사용률 평균 50% 이하, 메모리 사용 2GB 이하 (Apple Silicon Mac 기준). |
| 출처 | S-01 (암묵 — Mac 일반 사용 환경) |
| 우선순위 | 권장 |
| 검증 방법 | 1시간 세션 후 Activity Monitor 측정 |

### NFR-008 프라이버시
| 항목 | 내용 |
|---|---|
| 설명 | 세션 종료 시 메모리상 음성 데이터를 즉시 해제해야 한다. 외부 API 호출 시 본 사항이 사용자에게 명시되어야 한다. |
| 출처 | S-02 (회의 내용 민감성 — 암묵) |
| 우선순위 | 중요 |
| 검증 방법 | 코드 리뷰 + 첫 실행 시 외부 호출 고지 다이얼로그 확인 |
